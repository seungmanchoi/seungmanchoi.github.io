설계 결정: ReAct Agent vs Direct LLM
에이전트별 실행 방식을 다르게 선택한 이유

모든 에이전트는 ReAct Agent로 정의되어 있지만, 워크플로우 실행 시점에서 Insight Analyst, Chart Advisor, Followup Agent는 에이전트를 거치지 않고 Direct LLM 호출(model.invoke())을 사용합니다. 이 설계에는 세 가지 핵심 이유가 있습니다.

기준ReAct Agent (SQL/Search Expert)Direct LLM (Insight/Chart/Followup)
도구 반복 필요성 필수 — 스키마 조회 → RAG 검색 → SQL 실행의 다단계 도구 호출 루프가 핵심 불필요 — 이미 확보된 데이터를 받아 분석 결과 JSON만 생성하면 됨
병렬 실행 ReAct 내부의 순차적 도구 호출로 인해 병렬화 이점 감소 단일 프롬프트 → 단일 응답으로 Promise.allSettled 기반 진정한 병렬 처리 가능
출력 포맷 제어 도구 호출 메타데이터가 섞여 JSON 추출이 복잡 프롬프트에 JSON 스키마를 지정하면 파싱 가능한 구조화 응답을 직접 반환
응답 속도 도구 선택 → 실행 → 판단 사이클로 추가 레이턴시 발생 단일 LLM 호출로 오버헤드 최소화
핵심 원칙 — "도구 호출 반복이 필요한가?"가 설계 기준입니다. SQL Expert처럼 외부 시스템과 상호작용하며 다단계 추론이 필요한 에이전트는 ReAct 패턴을, 분석/시각화/질문 생성처럼 결정론적 변환 작업을 수행하는 에이전트는 Direct LLM을 사용하여 속도와 구조적 단순함을 확보합니다.
SQL Expert Agent
ReAct Pattern + RAG Enhanced

LangChain의 ReAct Agent를 사용하여 도구 기반으로 동작합니다. 스키마 조회 → 질문 임베딩으로 유사 질문 검색 → 매핑된 SQL 참조 → SQL 생성 → 실행의 자율 루프를 수행하며, SELECT 문만 허용하는 보안 제한이 적용됩니다.

RAG 연동 — get_similar_sql_examples 도구를 통해 사용자 질문을 임베딩하고, 유사 질문에 매핑된 SQL을 Top-3 검색하여 Few-Shot 컨텍스트로 활용, 생성 정확도를 크게 향상시킵니다.
병렬 분석 (Parallel Analysis)
Promise.allSettled 동시 실행

Insight Analyst와 Chart Advisor를 Promise.allSettled로 동시 실행하여 응답 시간을 단축합니다. 각 에이전트의 실행 시간이 독립적으로 기록됩니다.

const [insight, chart] = await Promise.allSettled([ insightModel.invoke([...]), // Insight 분석 chartModel.invoke([...]) // 차트 추천 ]); // 개별 에러 처리로 하나가 실패해도 나머지 정상 반환

Phase 2 — RAG 강화 및 가시성 확보
2025.12.06-08 | RAG 정확도 향상

  • 워크플로우 시각화 컴포넌트 도입 — 프론트엔드에서 진행률 표시

적용한 3가지 최적화

최적화변경 내용효과
병렬 실행 Insight + Chart를 Promise.allSettled로 동시 실행 분석 단계 소요 시간 50% 감소 (10-16s → 3-5s)
Direct LLM 전환 분석 에이전트를 ReAct → model.invoke() 직접 호출로 변경 도구 선택 오버헤드 제거, JSON 출력 포맷 안정화
모델 분리 Followup Agent에 Haiku 적용 (Sonnet → Haiku) 후속 질문 생성 3-5s → 1-2s, 토큰 비용 절감

Phase 4 — Query Learning 자동화
2025.12.13 | 실패 쿼리 자동 복구

문제

  • SQL 실행 실패 시 사용자에게 에러만 반환 — 재질의 필요
  • 실패 패턴이 반복되지만 학습되지 않음
  • RAG 지식 베이스가 수동으로만 확장 가능

해결

  • 최대 5회 자동 수정 시도 → EXPLAIN + LIMIT 1 검증
  • 성공 시 자동 RAG 임베딩 — 지식 베이스 자동 성장
  • 실패 쿼리 → failed_queries 인덱스에 저장하여 관리자 검토
  • 실패 복구율 70-80% 달성
